# HyperFocus OS War Room: Agent Command

## Command Rule
Use specialist agents when they materially reduce risk, improve clarity, or increase delivery speed. The goal is not to call every agent every time. The goal is to route the right work to the right specialist fast.

## Default Routing
- `broski-orchestrator`
  - Use for multi-part features, pack coordination, and cross-agent sequencing
  - Output: task breakdown, ownership, and next action
- `project-strategist`
  - Use for scope control, priority calls, MVP trimming, and competition strategy
  - Output: rankings, decision matrix, and scope boundaries
- `cto-system-architect`
  - Use for system shape, shared-core boundaries, and long-term technical direction
  - Output: architecture notes, module boundaries, and stack decisions
- `frontend-craftsman`
  - Use for hero screens, visual polish, demo UI, and frontend structure
  - Output: layout direction, components, and presentation choices
- `ux-flow-guardian`
  - Use for low-cognitive-load flows, empty states, rescue paths, and clarity
  - Output: user journey, core screens, and interaction simplification
- `backend-architect`
  - Use for APIs, domain services, data flow, and fastest credible implementation
  - Output: entities, route shape, service boundaries, and integration plan
- `security-auditor`
  - Use for privacy review, data boundaries, safe claims, and trust posture
  - Output: risks, forbidden claims, and mitigation rules
- `qa-test-engineer`
  - Use for demo-path testing, release gates, fallback modes, and failure analysis
  - Output: test matrix, QA gates, and verification plan
- `hyper-narrator`
  - Use for story arc, deck narrative, tagline work, and demo script tone
  - Output: pitch copy, story beats, and presentation language
- `deep-dive-analyst`
  - Use for market positioning, differentiation, and crowdedness checks
  - Output: competitor angle, positioning memo, and judge reaction analysis
- `manifest-enforcer`
  - Use for neurodivergent-first, privacy-focused, and performance alignment
  - Output: principle checks and course corrections

## Operating Rhythm
- Start with goal, scope boundary, and output needed.
- Route to one lead specialist for the active subtask.
- Pull in secondary specialists only where risk increases.
- Close each cycle with: what changed, what was checked, what is next.

## Default Work Modes
- Strategy mode
  - Lead: `project-strategist`
  - Support: `deep-dive-analyst`, `hyper-narrator`
- Product mode
  - Lead: `ux-flow-guardian`
  - Support: `frontend-craftsman`, `project-strategist`
- System mode
  - Lead: `cto-system-architect`
  - Support: `backend-architect`, `security-auditor`
- Launch mode
  - Lead: `qa-test-engineer`
  - Support: `security-auditor`, `broski-orchestrator`

## Escalation Triggers
- Scope is growing faster than confidence.
- The product starts to feel generic.
- The interface gets noisy or hard to explain.
- Privacy boundaries become fuzzy.
- The live demo path feels fragile.
- A decision affects multiple domains at once.

## Definition Of Done
- The assigned specialist produced a concrete output.
- A follow-up owner is clear.
- Risks are named, not hidden.
- The result improves the live path, not just the docs.
